# Multi-Encode Search 拡張機能 設計書

## 1. 概要

本ドキュメントは、VS Code / Cursor 向けの文字列検索拡張機能の設計をまとめたものです。

この拡張機能の主目的は、ファイルの文字コード差異による検索漏れを防ぐことです。

初期対応する文字コード:

- UTF-8
- Shift_JIS
- EUC-JP

初期状態で有効にする言語プリセット:

- 日本語
- 英語

配布形式は VS Code 互換拡張機能とし、OpenVSX で公開して Cursor からも利用できる構成を前提とします。

また、Mac / Windows の両環境で同一パッケージが動作することを前提に設計します。

## 2. 背景

一般的なエディタの文字列検索は UTF-8 前提であることが多く、レガシーな日本語資産を含むリポジトリでは Shift_JIS や EUC-JP のファイルが検索対象から実質的に漏れることがあります。

本拡張機能では以下を行うことで、その問題を解消します。

- ワークスペース内の候補ファイルを走査する
- 複数文字コードでファイルをデコードする
- 同一クエリを横断的に検索する
- 検索結果を 1 つの UI にまとめて表示する

## 3. ゴール

- UTF-8、Shift_JIS、EUC-JP を 1 回の検索で横断検索できる
- 将来的に他言語・他文字コードへ設定で拡張できる
- 大規模リポジトリでも実用的な速度を維持できる
- 除外ディレクトリ、除外ファイル名、除外拡張子を設定できる
- `.git` や `node_modules` など一般的な除外パターンを初期設定で有効化する
- VS Code と Cursor の両方で動作する
- Mac / Windows の両方で動作する
- OpenVSX に公開可能な構成にする

## 4. 非ゴール

- VS Code 標準検索 UI を完全に置き換えること
- v1 で永続インデックス型の全文検索基盤を作ること
- あらゆるファイルに対して厳密な自動文字コード判定を行うこと
- すべてのバイナリ形式を完全判定すること
- v1 で置換機能まで含めること

## 5. プロダクトスコープ

拡張機能は、独自コマンドから実行する検索機能として提供します。

- ユーザーは `Multi-Encode Search: Search in Workspace` のようなコマンドを実行する
- 検索文字列を入力する
- 拡張機能がワークスペース内の候補ファイルを列挙する
- 除外設定を適用する
- 各ファイルを複数文字コードで順次検索する
- 検索結果を専用ビューに一覧表示する
- 結果クリックで該当ファイルの該当行を開ける

## 6. 対応プラットフォーム

- VS Code
- Cursor
- macOS
- Windows
- OpenVSX 配布

実装は標準的な VS Code Extension API の範囲に収め、同一パッケージで両エディタに対応する方針とします。

OS 依存のシェルコマンドや外部 grep バイナリには依存せず、Node.js と VS Code API ベースで完結する設計にします。

## 7. 想定ユーザーストーリー

### 7.1 日本語レガシー資産の横断検索

- 日本語中心のレガシーコードベースを扱う開発者として、UTF-8 / Shift_JIS / EUC-JP を意識せず 1 回で検索したい

### 7.2 多言語リポジトリ対応

- 国際化されたコードベースを扱う開発者として、必要に応じて追加の文字コードや言語プリセットを有効化したい

### 7.3 速度制御

- 大規模リポジトリを扱うユーザーとして、初回表示件数やファイルサイズ上限を制御し、重くなりすぎないようにしたい

### 7.4 除外制御

- ユーザーとして、ノイズになりやすいディレクトリやファイルを除外して、必要な結果だけを見たい

## 8. 機能要件

### 8.1 検索入力

- ワークスペース検索を開始するコマンドを提供する
- ユーザーはプレーンテキストの検索語を入力できる
- 大文字小文字の区別は設定で制御できるようにする
- 正規表現検索は v1 では必須とせず、コスト次第で後続対応とする

### 8.2 複数文字コード検索

- 各候補ファイルに対して、設定済みの複数文字コードで検索を試行する
- v1 の初期有効文字コードは以下とする
  - `utf8`
  - `shift_jis`
  - `euc-jp`
- ある文字コードでマッチした時点で、そのファイルに対する残りの文字コード試行を打ち切る

補足:

- 将来、全ヒット文字コードを表示するモードは拡張要件として追加可能

### 8.3 言語プリセット

- 言語プリセットを設定として持てるようにする
- v1 では以下を初期有効化する
  - 日本語
  - 英語
- 他の言語は設定追加可能な構造にしておく

補足:

- 言語プリセットは実行時の厳密な言語判定ではなく、「どの文字コード群を候補に含めるか」の設定補助として扱う

### 8.4 ファイル除外

- 以下の単位で除外できるようにする
  - ディレクトリ名または glob
  - ファイル名または glob
  - 拡張子
- 初期除外候補には次を含める
  - `.git`
  - `node_modules`
  - `dist`
  - `build`
  - `coverage`
  - `.next`
  - `.env`
  - `.env.development.local`
  - `.env.test.local`
  - `.env.development`
  - `.env.test`
  - `.DS_Store`
  - `.vscode`
  - `.vscode/**`
  - `.idea`
  - `.idea/**`
  - `.cursor`
  - `.cursor/**`
- `.gitignore` を参照する設定を提供する

### 8.5 結果表示

- 検索結果には少なくとも以下を表示する
  - ファイルパス
  - 行番号
  - 該当行のプレビュー
  - ヒット時に使われた文字コード
- ファイル単位でグルーピングして表示する
- クリックで該当箇所を開ける

### 8.6 軽量化・制御項目

- 初回表示ファイル数に上限を持たせられる
- 以下の制御項目を設定化する
  - 検索対象ファイルの最大サイズ
  - ファイルごとの最大ヒット数
  - 全体の最大ヒット数
  - 並列処理数
- 可能な範囲でバイナリファイルを事前スキップする

## 9. 非機能要件

### 9.1 性能

- 検索開始後、できるだけ早く最初の結果を返す
- 全ファイルを一括読み込みせず、順次処理する
- 並列処理は上限付きにする

### 9.2 信頼性

- 一部ファイルのデコード失敗で検索全体を止めない
- 個別ファイルの読み込み失敗はログ出力しつつ継続する

### 9.3 互換性

- 安定した VS Code API のみを利用する
- macOS / Windows のパス差異を吸収できる実装にする
- OS 依存のコマンドやファイルシステム前提を持ち込まない
- OpenVSX 公開可能なパッケージ構成にする

## 10. 全体アーキテクチャ

以下のモジュール構成を想定します。

### 10.1 Command Layer

責務:

- コマンド登録
- 検索入力の受け付け
- 検索開始
- 検索キャンセル

### 10.2 Configuration Layer

責務:

- `settings.json` から設定を読み込む
- デフォルト設定を解決する
- ユーザー設定をマージする

### 10.3 File Discovery Layer

責務:

- ワークスペース内の候補ファイル列挙
- 除外設定適用
- `.gitignore` 反映
- サイズ上限によるスキップ

実装方針:

- まずは VS Code の Workspace API か軽量な走査実装を使う
- 必要に応じて glob ライブラリ導入を検討する
- パス結合や正規化は Node.js の `path` 系 API を通して行う
- 内部比較用パスは区切り文字を正規化して扱う

### 10.4 Search Engine Layer

責務:

- ファイルを `Buffer` として読み込む
- バイナリらしきファイルを弾く
- 有効な文字コード順にデコードする
- 検索語を探す
- 行単位の結果に整形する

推奨ライブラリ:

- `iconv-lite`

補足:

- 文字コード変換はライブラリベースで完結させ、OS 標準コマンドやロケール依存挙動に頼らない

### 10.5 Result Presentation Layer

責務:

- 検索結果状態の保持
- `TreeView` などによる結果表示
- クリック時のファイルオープン

## 11. 検索フロー

1. ユーザーが検索コマンドを実行する
2. 拡張機能が検索文字列を受け付ける
3. 現在設定を読み込む
4. 候補ファイルを列挙する
5. 除外条件とサイズ条件を適用する
6. 上限付き並列でファイルを処理する
7. 各ファイルについて以下を行う
   - 生バイト列を読む
   - バイナリらしき場合はスキップする
   - 優先順に文字コードを試す
   - デコード後文字列に対して検索する
   - マッチした結果を格納する
8. 結果ビューへ逐次反映する
9. 上限到達またはキャンセル時に終了する

## 12. 文字コード戦略

### 12.1 初期優先順

初期の試行順は以下とします。

1. UTF-8
2. Shift_JIS
3. EUC-JP

理由:

- UTF-8 は現代的なリポジトリで最も一般的
- Shift_JIS は日本語レガシー資産で頻出
- EUC-JP も一部の旧資産では依然として残る

### 12.2 マッチ判定

v1 では以下を満たした場合にヒットとみなします。

- デコード後に検索可能な文字列を得られる
- その文字列に検索語が含まれる

留意点:

- 一部のバイト列は複数文字コードでそれらしくデコードされる可能性がある
- v1 では「優先順で最初にヒットした文字コード」を採用する

### 12.3 将来改善

- デコード妥当性のスコアリング
- 拡張子別の文字コードヒント
- ファイル単位の推定結果キャッシュ

## 13. 性能設計

### 13.1 基本方針

- まず最初の有用な結果を速く返す
- v1 では事前インデックスを持たない
- 不要なデコード試行を減らす

### 13.2 具体策

- 候補列挙と検索処理をできる限り逐次化または小分けにする
- 並列数に上限を持たせる
- 大きすぎるファイルを初期設定でスキップする
- 1 つの文字コードでヒットしたら他文字コードを試さない
- 初回表示件数で止め、必要なら将来 `Load More` を追加する
- 除外パターンは事前コンパイルして再利用する
- OS ごとの I/O 差異を考慮し、並列数は固定値だけでなく設定変更可能にする

### 13.3 推奨デフォルト値

v1 の初期値候補:

- 初回表示対象ファイル数: 200
- ファイルごとの最大ヒット数: 20
- 最大ファイルサイズ: 2 MB
- 並列数: 6

これらはすべて設定で変更可能にします。

## 14. デフォルト設定案

```json
{
  "multiEncodeSearch.enabledEncodings": [
    "utf8",
    "shift_jis",
    "euc-jp"
  ],
  "multiEncodeSearch.languagePresets": {
    "japanese": true,
    "english": true
  },
  "multiEncodeSearch.excludeGlobs": [
    "**/.git/**",
    "**/node_modules/**",
    "**/dist/**",
    "**/build/**",
    "**/coverage/**",
    "**/.next/**"
  ],
  "multiEncodeSearch.excludeExtensions": [
    ".png",
    ".jpg",
    ".jpeg",
    ".gif",
    ".pdf",
    ".zip",
    ".exe",
    ".dll"
  ],
  "multiEncodeSearch.useGitIgnore": true,
  "multiEncodeSearch.maxInitialMatchedFiles": 200,
  "multiEncodeSearch.maxMatchesPerFile": 20,
  "multiEncodeSearch.maxFileSizeBytes": 2097152,
  "multiEncodeSearch.concurrency": 6,
  "multiEncodeSearch.caseSensitive": false
}
```

## 16. UI 設計

### 16.1 コマンド

初期コマンド案:

- `Multi-Encode Search: Search in Workspace`
- `Multi-Encode Search: Cancel Search`
- `Multi-Encode Search: Clear Results`

### 16.2 結果ビュー

推奨方針:

- 専用 View Container またはパネルビューを追加する
- ファイル単位でグルーピングして表示する
- 各ファイルまたは各マッチに文字コードラベルを出す

表示イメージ:

- `src/legacy/foo.txt [shift_jis] (3 matches)`

### 16.3 UX 補足

- 長時間検索時は進捗表示を出す
- キャンセル可能にする
- 検索中でも結果ビューは逐次更新する
- パス表示は各 OS の見た目に配慮しつつ、内部処理では正規化済みパスを利用する

## 17. エラーハンドリング

- 文字コードごとのデコード失敗は通常フローとして扱い、ユーザーへ過剰通知しない
- ファイル読み込み失敗は Output Channel に記録する
- ワークスペース全体に関わる致命的エラーのみ簡潔に通知する

## 18. テレメトリ・プライバシー

v1 方針:

- デフォルトでテレメトリは送信しない
- 検索処理に外部ネットワークアクセスは不要

将来テレメトリを追加する場合も、明示的な opt-in を前提とします。

## 19. OpenVSX 公開設計

### 19.1 パッケージ方針

- 標準的な VS Code 拡張機能構成でパッケージ化する
- OpenVSX 公開要件を満たすメタデータを整える

### 19.2 公開前提のメタ情報

- publisher 名と extension ID は早期に確定する
- 以下を公開前に準備する
  - `README.md`
  - `LICENSE`
  - `CHANGELOG.md`
  - アイコン

### 19.3 互換性方針

- 可能な限り 1 パッケージで VS Code / Cursor 両対応とする
- 可能な限り 1 パッケージで macOS / Windows 両対応とする
- エディタ別 fork は互換性問題が出た場合にのみ検討する

### 19.4 クロスプラットフォーム方針

- `rg` や `grep` など OS 環境依存の外部コマンドには依存しない
- 改行コードは `LF` と `CRLF` の両方を扱えるようにする
- パス比較時は区切り文字と大文字小文字差異に注意する
- Windows 特有の長いパスやドライブレター表現を考慮した実装にする
- ファイル権限エラーや共有ロックなど、Windows で起きやすい読み取り失敗は個別エラーとして握りつぶさずログ記録して継続する

## 20. リスク

### 20.1 あいまいなデコード

リスク:

- 同一バイト列が複数文字コードでそれらしく読めてしまい、誤判定が起きる可能性がある

対策:

- 優先順を明示する
- 結果に採用文字コードを表示する
- 使わない文字コードは設定で無効化できるようにする

### 20.2 大規模リポジトリでのコスト増

リスク:

- 非常に大きなリポジトリでは時間やメモリ使用量が増える

対策:

- 除外条件を早期適用する
- ファイルサイズと件数に上限を設ける
- 並列数を制限する

### 20.3 API 制約

リスク:

- VS Code API 上、標準検索パネルへの完全統合は難しい可能性がある

対策:

- 標準検索の置き換えではなく、専用検索 UI として設計する

### 20.4 OS 差異による不具合

リスク:

- パス区切り文字、改行コード、ファイルロック挙動、大文字小文字の扱い差で検索漏れや誤除外が起きる可能性がある

対策:

- パスは内部で正規化する
- 改行コードに依存しない行分割処理を使う
- Windows で発生しやすい読み取り例外を個別処理する
- Mac / Windows の両方で最低限の動作確認を行う

## 24. 推奨判断

まずは以下の方針で MVP を進めるのが現実的です。

- 独自検索コマンド方式
- 専用結果ビュー方式
- `iconv-lite` によるデコード
- ユーザー設定可能な除外パターン
- 保守的な初期上限による軽量化

この構成なら、要件の中心である「検索漏れ防止」と「軽さ」を両立しつつ、OpenVSX 公開まで見据えた現実的な第一版を作れます。
